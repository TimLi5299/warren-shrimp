# 经验教训知识库

> **"聪明人从自己的错误中学习，更聪明的人从别人的错误中学习。"**  
> —— 奥托·冯·俾斯麦

---

## 📚 这是什么？

这是一个**持续成长的经验教训知识库**，记录每次交付结果与预期不一致时的：

1. **问题描述** - 发生了什么
2. **根本原因** - 为什么会发生
3. **解决方案** - 如何修复
4. **核心教训** - 学到了什么
5. **可复用模式** - 如何应用到未来

---

## 🎯 目标

- ✅ **避免重复犯错** - 同样的问题不出现第二次
- ✅ **加速学习曲线** - 新项目的启动有章可循
- ✅ **沉淀最佳实践** - 经验转化为可复用的工具和模板
- ✅ **持续改进** - 每次问题都是改进的机会

---

## 📁 知识库结构

```
lessons-learned/
├── README.md              # 本文件
├── 001-case-library-links-missing.md  # 教训 #001
├── 002-xxx.md             # 未来的教训...
└── index.json             # 索引（可选，用于快速检索）
```

---

## 🏷️ 教训编号规则

格式：`NNN-short-description.md`

- `NNN` - 三位数序号（001, 002, 003...）
- `short-description` - 简短描述（kebab-case）

示例：
- `001-case-library-links-missing.md` - 案例库链接缺失
- `002-deployment-without-testing.md` - 未测试就部署
- `003-hardcoded-paths.md` - 硬编码路径

## 📝 记录模板

每个教训文件包含：

```markdown
# 教训 #NNN：简短标题

## 📅 元信息
- 发现日期
- 项目
- 问题类型
- 严重程度

## 🔍 问题描述
现象、用户提问、实际状态

## 🧠 根本原因分析
直接原因、深层原因、思维盲区

## 💡 解决方案
短期修复、长期预防

## 🎯 核心教训（可复用）
抽象为通用原则

## 🔄 可复用的模式
适用场景、通用工具

## 📝 如何应用到未来项目
检查清单、快速启动

## 📊 影响评估
不修复的后果、修复后的收益

## 🔗 相关文件
相关配置文件、脚本、文档

## 🏷️ 标签
#标签 #用于 #分类 #检索
```

## 🔍 如何使用

### 遇到问题时

1. **先搜索** - 查看是否已有类似教训
   ```bash
   grep -r "关键词" lessons-learned/
   ```

2. **参考解决方案** - 应用已有的最佳实践

3. **如无匹配** - 解决后创建新教训文件

### 开始新项目时

1. **浏览教训列表** - 了解历史问题

2. **应用检查清单** - 每个教训的"如何应用到未来项目"部分

3. **复制通用工具** - 使用已抽象的框架和模板

## 💬 文化原则

### ✅ 鼓励的

- **坦诚记录** - 不回避问题，不掩盖错误
- **深度分析** - 找到根本原因，不只是表面症状
- **积极分享** - 教训是财富，不是负担
- **持续应用** - 学到的东西要用起来

### ❌ 避免的

- **指责文化** - 对事不对人
- **表面文章** - 不要为了记录而记录
- **束之高阁** - 记录了就要用
- **重复犯错** - 同样的问题不出现第二次
