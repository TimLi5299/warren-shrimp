## 组件 1：玩家位卡（PlayerCard）

玩家位卡是解决 **P0（回合状态不可感知）** 和 **P2（阵营感知缺失）** 的主战场。

### 1.1 基础结构

```
┌─────────────────────────────┐
│   [头像圆形]  [昵称]  [牌数]  │
│       ↑                     │
│   回合光晕层（绝对定位）       │
└─────────────────────────────┘
```

- **头像**：直径 `40px`，`radius-avatar`（50%），背景色用 `surface-panel`
- **昵称**：`type-player-name`（13px/500），颜色 `text-secondary`
- **手牌计数徽章**：高度 `20px`，横向 `space-xs`（4px）内边距，`radius-badge`（pill），`type-badge`（11px/700）

### 1.2 阵营版本差异

这是最关键的区分规则。**无论任何状态，阵营色必须持续可见**，不因非激活而完全消失。

| 属性 | 队友（对家） | 对手（左家/右家） |
|------|------------|----------------|
| 头像边框（静止态） | `2px solid faction-ally-muted`（`#9A6820`） | `2px solid faction-opp-muted`（`#2D5980`） |
| 头像边框（激活态） | `3px solid faction-ally-primary`（`#E8A230`） | `3px solid faction-opp-primary`（`#4B8FC4`） |
| 牌数徽章底色（静止态） | `faction-ally-muted`（`#9A6820`） | `faction-opp-muted`（`#2D5980`） |
| 牌数徽章底色（激活态） | `faction-ally-primary`（`#E8A230`） | `faction-opp-primary`（`#4B8FC4`） |
| 徽章文字色 | `text-on-primary-btn`（深色，在金底上） | `text-primary`（浅色，在蓝底上） |

> **为什么静止态也保留 muted 阵营色而非统一灰色？**  
> 如果非激活态统一用灰色，玩家每次回合切换时需要重新辨认"哪个是队友哪个是对手"。保留 muted 色使阵营信息成为"背景知识"而非"每次都要推断"——就像棋盘上黑子和白子永远是黑白，无论当前谁走棋。

### 1.3 回合状态差异（解决 P0 核心）

| 状态 | 视觉表现 | Token 来源 |
|------|---------|-----------|
| 非当前回合（静止） | 普通边框（muted 阵营色）+ `shadow-avatar-idle` | §2.2 + §5.2 |
| 当前回合（我方） | 边框加粗至 3px + `shadow-avatar-ally-active` + 脉冲 opacity 动效 | 动效 §6.3D |
| 当前回合（对手） | 边框加粗至 3px + `shadow-avatar-opp-active` + 脉冲 opacity 动效 | 动效 §6.3D |
| AI 思考中 | 在激活光晕基础上，昵称旁显示 "…" 动态省略号（3点循环，`duration-normal`×3 周期） | — |
| 已打完（胜出/清牌） | 头像叠加半透明 `surface-panel` 遮罩 + "✓" 图标，光晕消失 | 语义 `semantic-success` |

### 1.4 四个座位的空间分布

| 座位 | 位置 | 头像朝向 | 组件宽度 |
|------|------|---------|---------|
| 我（下） | 不显示 PlayerCard，由底部手牌区和按钮组承载身份 | — | — |
| 对家（上，队友） | 顶部中央，固定居中 | 朝下 | `160px` |
| 左家（对手） | 左侧中部，垂直居中 | 朝右 | `120px`（紧凑模式，旋转 90°） |
| 右家（对手） | 右侧中部，垂直居中 | 朝左 | `120px`（旋转 -90°） |

> 左右玩家位卡建议旋转 90°（垂直布局），符合围桌坐向的空间直觉，同时释放中央舞台的横向空间。

## 组件 2：手牌区（HandArea）

手牌区是解决 **P1（手牌密度过高）** 的主要组件。

### 2.1 卡牌基础尺寸

| 属性 | 值 | 说明 |
|------|----|----|
| 卡牌宽度（静止） | `48px` | 在 1440px 宽、27 张牌下，重叠后每张露出 32px，总占宽约 864px，留有边距 |
| 卡牌高度（静止） | `72px` | 宽高比 2:3，符合标准扑克比例 |
| 卡牌宽度（焦点/放大） | `80px` | 悬停时 scale 等效值（实现为 scale(1.08) + 自然扩张不做实际宽度变化，避免布局回流） |
| 卡牌高度（焦点/放大） | `115px` | 悬停放大后的等效高度 |
| 重叠量（cards overlap） | `-16px`（负 margin-right） | 使用 `space-md`（16px）的负值；每张牌露出 32px |
| 卡牌圆角 | `radius-card`（6px） | — |

### 2.2 四态视觉规范

| 状态 | 触发条件 | 视觉变化 |
|------|---------|---------|
| **静止态** | 默认 | `shadow-card-rest`；`translateY(0)`；`opacity: 1` |
| **悬停态** | `mouseover` | `translateY(-10px)`；`scale(1.08)`；`shadow-card-hover`；`duration-fast`（150ms）；`ease-out-enter` |
| **选中态** | 单击 | `translateY(-20px)`；`shadow-card-selected`（含金色描边 + 光晕）；`duration-normal`（250ms）；`spring-feedback` |
| **不可出牌态** | 游戏规则判定不可出 | `opacity: 0.40`；`filter: saturate(0.3)`；无 hover 响应；`cursor: not-allowed` |

> **为什么悬停和选中分别用 10px 和 20px 的不同位移？**  
> 悬停是"预览意图"，需要有响应但不能假装已选中——10px 刚好越过视觉噪声阈值（玩家能感知到变化），但不至于与选中态混淆。选中的 20px 是一个明确的承诺性位移，配合金色描边，玩家不会误以为"是悬停还是选中了"。两个值的比例 1:2 构建了清晰的语义层次。

> **为什么不可出牌用 `saturate(0.3)` 而非仅用 `opacity: 0.40`？**  
> 纯透明度降低使牌面变成"半透明叠层"，仍保留色彩，玩家容易误以为"颜色看不清"而非"这张牌不能出"。同时降低饱和度使牌面趋近灰色，在视觉语言上更接近"灰掉/禁用"的普世隐喻。

### 2.3 手牌排列规则

- 默认以红桃/方块/梅花/黑桃花色分组，同花色内按点数升序排列（从左到右低到高）
- 选中牌上移后，两侧相邻牌补充 `±4px` 横移，给选中牌"留出呼吸空间"（可选优化）
- 手牌数量 ≤ 5 张时，取消重叠，改为 `space-lg`（24px）间距展开，每张牌完整可见
- 手牌区底部有 `space-xl`（32px）的空白区域将其与按钮组隔开

## 组件 3：桌面中央出牌区（CenterPlayZone）

这个组件直接解决 **P1（中央区常驻空白）** 问题。当前版本中央出牌区是"虚空"，新规范赋予它永久存在的视觉锚。

### 3.1 空舞台状态（常驻视觉锚）

无任何出牌时，中央舞台必须显示以下"视觉锚"：

| 属性 | 规格 |
|------|------|
| 底色 | `surface-felt-stage`（`#1E5435`），比主桌面亮约 8% |
| 形状 | 圆角矩形，宽 `480px`，高 `220px`，`radius-stage-hint`（8px） |
| 边框 | `1px dashed rgba(255,255,255,0.10)`——仅有虚线轮廓，不填实色框 |
| 中央文字 | "等待出牌"（或为空），`type-system-msg`（13px/400），`text-muted`（`#5A7060`） |
| 位置 | 桌面中央，垂直居中偏上 20px（视觉重心的物理偏移） |
| 出现动效 | 在游戏开始后 `duration-slow`（400ms）fade in，不在大厅页显示 |

> **为什么用虚线框而非实线框？**  
> 实线框赋予舞台区过强的"容器"语义，玩家会下意识地期待框内装东西，增加焦虑感。虚线框是"可能性"的视觉语言——这里是行动会发生的位置，但目前还在等待。这种区分在 UI 语言中是约定俗成的（参考表单的 drag-drop 区域）。

### 3.2 出牌后状态

当某位玩家出牌，牌组以飞行动效（§6.3C）到达后：

| 属性 | 规格 |
|------|------|
| 牌组位置 | 根据出牌方向分布在舞台中心的四个象限（见下表） |
| 牌组排列 | 最新一手牌扇形展开（最多展示 5 张，超出折叠），`shadow-card-rest` |
| 出牌方标注 | 牌组下方显示出牌方昵称，`type-hint`（11px/400），对应阵营色的 muted 版本 |
| 显示时长 | `duration-linger`（2500ms），之后 fade out（`duration-normal`，250ms） |
| 下家出牌后 | 前一手牌立即开始淡出，不等满 2500ms |

**四方向牌组分布（桌面中央坐标系）：**

| 出牌方 | 在舞台区的落点 |
|--------|-------------|
| 我（下家，South） | 舞台区下半部，水平居中 |
| 对家（上家，North） | 舞台区上半部，水平居中 |
| 左家（West） | 舞台区左半部，垂直居中 |
| 右家（East） | 舞台区右半部，垂直居中 |

> 这个四象限布局让玩家不看名字标注，仅凭牌组的空间位置就能立刻判断"谁刚出的牌"，强化了空间语义与座位关系的绑定。

### 3.3 上家出牌展示（重要）

"上家刚出的牌"需要比其他玩家的牌组有更高的视觉优先级，因为这是玩家判断"能否跟牌"的参照物：

- 上家牌组保持 `opacity: 1`，其他牌组降至 `opacity: 0.55`
- 上家牌组额外有 `shadow-card-hover` 层级的阴影（比普通出牌记录更"立体"）
- 当轮到玩家出牌时，如上家牌组仍在显示，其标注文字更新为"需跟过的牌型"字样

## 组件 4：底部操作按钮组（ActionButtonGroup）

解决 **P1（主 CTA 不突出）** 和 **P2（按钮语义混乱）**。

### 4.1 三级视觉权重体系

**核心规则：一眼扫过去，玩家的视线应该按照「出牌 → 不出 → 其他」的顺序依次注意到三组按钮。**

| 层级 | 按钮 | 视觉权重 |
|------|------|---------|
| **主CTA** | 出牌 | 最大、最亮、有填充色 |
| **次要操作** | 不出 | 中等、有填充色但较暗 |
| **工具操作** | 提示、理牌、广播 | 最小、透明填充、视觉沉入背景 |

### 4.2 主CTA：出牌按钮

| 属性 | 规格 | Token 来源 |
|------|------|-----------|
| 宽度 | `120px` | 固定宽，不随文字变化 |
| 高度 | `48px` | 最高，视觉重量最大 |
| 背景色 | `semantic-primary`（`#E8A230`） | §2.3 |
| 文字 | "出牌"，`type-btn-primary`（15px/700/0.5px字距） | §4 |
| 文字色 | `text-on-primary-btn`（`#1A2E1A`）—— 深色文字在金底上 | §2.5 |
| 圆角 | `radius-btn-primary`（10px） | §5.1 |
| 阴影（静止） | `shadow-btn-rest` | §5.2 |
| 禁用态（未选牌时） | `opacity: 0.50`；`cursor: not-allowed`；色彩不变（保持金色标识，告知玩家"选了牌就能用"） | — |
| 禁用态的说明 | 不改变颜色的原因：让玩家一直看到"金色=出牌"的关联，选完牌后直接触发反馈而不需要"发现按钮变色了" | — |

### 4.3 次要操作：不出按钮

| 属性 | 规格 | Token 来源 |
|------|------|-----------|
| 宽度 | `96px` | 比出牌小 20% |
| 高度 | `40px` | 比出牌低 8px |
| 背景色 | `semantic-secondary`（`#3D6B8A`） | §2.3 |
| 文字 | "不出"，`type-btn-secondary`（14px/600） | §4 |
| 文字色 | `text-primary`（`#EEF2EE`） | §2.5 |
| 圆角 | `radius-btn-secondary`（8px） | §5.1 |
| 隐藏逻辑 | 当规则强制必须出牌时（首出轮），此按钮 `display: none`——不留占位，让主CTA居中强调 | — |

### 4.4 工具操作：提示 / 理牌 / 广播

| 属性 | 规格 |
|------|------|
| 宽度 | `80px` |
| 高度 | `32px` |
| 背景色 | `semantic-tertiary`（`rgba(255,255,255,0.08)`） |
| 边框 | `1px solid semantic-tertiary-border`（`rgba(255,255,255,0.18)`） |
| 文字 | `type-btn-tertiary`（13px/400），`text-on-tertiary-btn` |
| 圆角 | `radius-btn-tertiary`（6px） |

> 工具按钮不使用橙色。广播按钮的功能是"通知/表情"，是辅助社交功能，应该视觉上"不打扰"——工程师看到橙色提示"重要操作"，但广播在大多数局面下不重要。移入 tertiary 风格，让玩家在关键出牌时刻自然忽略它，在想用时知道在哪里找。

### 4.5 按钮组布局

```
[理牌] [提示]        [不出] [出牌]
←── 工具区 ──→  ←── 操作区 ──→
```

- 整体右对齐（操作区贴右）：玩家的"决策动作"（出牌）在视觉上压向右侧，与鼠标/手指的自然右移轨迹一致
- 工具区和操作区之间有 `space-xl`（32px）间距，形成视觉分组
- 当不出按钮隐藏时，出牌按钮右对齐不变（工具区在左，出牌在右）
- 整个按钮组距底边 `space-lg`（24px）
- 按钮组距手牌区 `space-xl`（32px）——这个间距解决「手牌与操作混为一平面」的问题

### 4.6 按钮组激活/禁用与回合联动

| 回合状态 | 按钮组整体表现 |
|---------|-------------|
| 非我回合 | 整体 `opacity: 0.45`；pointer-events: none；不显示 hover 效果 |
| 我回合，未选牌 | 整体 `opacity: 1.0`；出牌按钮禁用（见 §4.2）；不出/工具可用 |
| 我回合，已选牌 | 整体 `opacity: 1.0`；出牌按钮激活 |
| 切换至我回合时 | 按钮组从 `opacity: 0.45 → 1.0` 在 `duration-normal`（250ms）内淡入 |

> **为什么非我回合用 0.45 而非完全 `display: none`？**  
> 保持按钮可见但灰化，让玩家在非回合时段仍然知道"按钮在哪里"，当回合切换时不需要寻找按钮，视线可以直接在预知位置完成操作。`display: none` 会导致按钮出现时的布局跳动（layout shift），破坏界面稳定性。

## 组件 5：顶部信息栏（TopInfoBar）

解决 **P2（关键 HUD 信息缺失）** 和审查报告中提到的"返回链接侵入游戏空间"问题。

### 5.1 信息栏结构

```
┌──────────────────────────────────────────────────────────────────────┐
│  [≡ 菜单]    [我方: ♥5]  [第 2 局]  [对方: ♦7]    [局时 02:47]      │
└──────────────────────────────────────────────────────────────────────┘
```

- 高度：`44px`
- 背景：`surface-panel`（`rgba(10, 28, 18, 0.88)`），`backdrop-filter: blur(8px)`
- 底边：`1px solid surface-panel-border`（`rgba(255,255,255,0.10)`）
- 不使用 `radius-panel`（保持矩形，紧贴顶边，与页面边缘无缝）
- 始终在游戏进行中保持可见，不在大厅/房间页显示

### 5.2 五个信息分区

| 分区 | 内容 | 样式 |
|------|------|------|
| **① 菜单入口** | "≡" 汉堡图标 | 宽 `44px`，居左，替代原来的"← 返回"悬浮链接 |
| **② 我方积分** | "我方" label + 当前级牌（如 ♥5） | `type-hud-label`（11px/500/0.8px字距）+ `type-hud-value`（14px/700）；`faction-ally-primary`（金）标注 |
| **③ 局数** | "第 X 局" | 居中，`type-hud-value`（14px/700），`text-secondary` |
| **④ 对方积分** | "对方" label + 当前级牌（如 ♦7） | 同 ②，但用 `faction-opp-primary`（蓝）标注 |
| **⑤ 局时** | 本局用时（MM:SS） | 居右，`type-hud-label`，`text-muted`；倒计时模式时变 `semantic-warning` |

> **为什么局时是次要信息（用 text-muted）？**  
> 掼蛋没有严格的计时规则，局时是"参考信息"而非"压力信息"。将其视觉降权，避免让玩家在策略思考时感到不必要的时间焦虑。只有在即将触发超时提示时才升级颜色（变为 `semantic-warning`），这个颜色升级本身就成为了超时警告的信号。

### 5.3 菜单展开内容（替代悬浮返回链接）

菜单按钮（汉堡图标）展开后显示：
- 返回大厅（等同于原"← 沃伦·巴菲虾"的功能）
- 游戏设置（音效开关等）
- 规则说明

菜单面板使用 `surface-panel` 背景 + `radius-panel`（12px），从左侧滑入，`duration-slow`（400ms）+ `ease-out-enter`。

**这将"返回链接侵入游戏空间"的问题彻底解决**：游戏内容区不再有任何来自产品层的导航元素直接覆盖。

### 5.4 信息栏与游戏状态的联动

| 游戏状态 | 顶栏变化 |
|---------|---------|
| 大厅/房间等待页 | `display: none`，不占空间 |
| 游戏进行中 | 完整显示，`opacity: 1` |
| 游戏结束/结算 | 局数更新；积分升级时对应分区有 `semantic-success` 短暂高亮（1000ms，然后恢复） |
| 升级（打完一轮） | 级牌数值更新，配合向上的数字滚动动效（`translateY(-100% → 0)`，`duration-normal`） |

## 附：组件间协作矩阵

确保五个组件在同一时刻的视觉信号不相互冲突：

| 当前游戏事件 | PlayerCard | HandArea | CenterPlayZone | ActionButtons | TopInfoBar |
|------------|-----------|----------|----------------|---------------|-----------|
| 我出牌 | 我的光晕保持激活 | 选中牌开始飞行动效 | 牌组飞入我方象限 | 整体短暂 disable（150ms防重复点击） | 无变化 |
| 对手出牌 | 对手光晕激活 + 脉冲 | 全牌组 opacity 0.45 | 牌组飞入对应象限 | 整体降至 opacity 0.45 | 无变化 |
| 回合切换到我 | 前一激活方光晕淡出 → 我的光晕淡入 | opacity 回 1.0 | 上家牌组保持高亮 | 从 0.45 淡入 1.0（250ms） | 无变化 |
