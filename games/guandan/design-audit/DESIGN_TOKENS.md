# 掼蛋游戏 · 设计 Token 系统

> 版本：v1.0 · 2026-05-01  
> 性质：设计宪法，所有视觉决策的唯一依据。工程师改样式时，优先从本文档取值，不自行发明数字。

---

## 一、设计定位

### 1.1 视觉气质：三个形容词

**沉浸 · 精密 · 有质感**

- **沉浸**：玩家坐在桌前时应该感觉"进入了一场真实的牌局"，而不是"打开了一个 Web 应用"。光源、投影、毡布纹理都是构建沉浸感的工具，不是装饰。
- **精密**：掼蛋是一款需要精密计算的战略游戏。UI 的每一个状态变化都应该是一次清晰的信息传递，不含糊，不模棱两可——谁的回合、谁刚出了什么牌、我能不能跟，这三件事在任意时刻都应该用视觉语言直接回答。
- **有质感**：卡牌应该有纸张感，桌面应该有毡布感，按钮按下应该有物理弹簧感。质感靠阴影、高光、运动曲线来塑造，而不是靠贴图。

### 1.2 目标情绪与视觉承托

玩家在这局牌的情绪弧线是：**期待（发牌）→ 分析（持牌）→ 紧张（关键回合）→ 释然或懊悔（结束）**。

视觉设计要服务于"分析"和"紧张"这两段——这是玩家停留时间最长、认知负荷最高的阶段。承托方式：
- 用低饱和的深绿底色降低背景噪音，让牌面信息成为最清晰的前景
- 用精确的状态颜色（回合指示器、阵营色）取代视觉上的静默，让"紧张时刻"在视觉上自然升温
- 避免无意义的装饰动效——所有动效必须携带信息

### 1.3 一句话设计原则

> 这款游戏看起来应该像**一张被认真对待的竞技牌桌**，而不是像**一个贴了扑克贴纸的 todo list**。

对标定位：
- 比天天斗地主**更克制**：去除卡通角色、气泡广告、过度动效
- 比 JJ掼蛋**更有氛围**：加入物理投影、毡布纹理感、阵营色系统
- 向 Balatro 借鉴**光源统一性**：所有阴影方向一致（正上方光源），形成视觉深度

## 二、色彩系统

### 2.1 基底色（Surface）

| Token | Hex | 用途 | 禁止用在 |
|-------|-----|------|---------|
| `surface-felt-main` | `#1A4A2E` | 桌面主色（全局背景） | 任何文字、图标的背景（对比度不足） |
| `surface-felt-stage` | `#1E5435` | 桌面中央出牌舞台区（比主色亮 8%） | 手牌区背景 |
| `surface-felt-edge` | `#14382200` | 桌面边缘渐变终点（径向渐变压暗） | 直接使用（仅用于渐变） |
| `surface-card` | `#F8F6F0` | 卡牌底色 | 任何非卡牌的面板/容器 |
| `surface-panel` | `rgba(10, 28, 18, 0.88)` | HUD 面板、信息栏、弹窗背景 | 大面积铺底（仅用于叠加层） |
| `surface-panel-border` | `rgba(255, 255, 255, 0.10)` | 面板边框描边 | 卡牌、按钮边框 |

> **为什么 `surface-felt-main` 选 `#1A4A2E` 而非更亮的绿？**  
> 截图中的原始桌面约为 `#1E6B3C`，饱和度过高，导致白色卡牌在其上显出轻微的品红色偏移（绿色补色效应）。`#1A4A2E` 将明度压低约 15%、饱和度降低约 20%，既保留毡布的绿色语义，又消除了补色视觉噪声，同时为所有前景元素提供更充足的对比度余量。

> **为什么 `surface-card` 选 `#F8F6F0` 而非纯白 `#FFFFFF`？**  
> 纯白与深绿产生高达 15:1 的极端对比度，在长时间游戏后会造成视觉疲劳，且让卡牌看起来像发光的屏幕而非纸张。`#F8F6F0` 加入极轻的暖黄偏移（6% 暖）模拟纸质质感，对比度降至约 11:1，仍远超 WCAG AA 标准（4.5:1）。

### 2.2 阵营色（Faction）⭐

阵营色是解决 **P2（阵营感知缺失）** 的核心 Token。在掼蛋中，两个对手（左家、右家）和一个队友（对家）的视觉区分度直接影响策略决策速度。

#### 我方 / 队友色（暖琥珀系）

| Token | Hex | 用途 |
|-------|-----|------|
| `faction-ally-primary` | `#E8A230` | 队友头像边框、我的回合指示器、出牌方向箭头 |
| `faction-ally-highlight` | `#F5C060` | 回合激活时的光晕外圈、悬停高亮 |
| `faction-ally-muted` | `#9A6820` | 非激活状态下的队友标识（弱化但保留阵营辨识） |
| `faction-ally-glow` | `rgba(232, 162, 48, 0.35)` | 回合指示光晕（box-shadow 外层扩散色） |

#### 对手色（冷钢蓝系）

| Token | Hex | 用途 |
|-------|-----|------|
| `faction-opp-primary` | `#4B8FC4` | 对手头像边框、对手回合指示器 |
| `faction-opp-highlight` | `#6AAED9` | 对手回合激活光晕 |
| `faction-opp-muted` | `#2D5980` | 非激活态对手标识 |
| `faction-opp-glow` | `rgba(75, 143, 196, 0.30)` | 对手回合光晕扩散色 |

#### 中立 / 系统色

| Token | Hex | 用途 |
|-------|-----|------|
| `faction-neutral-primary` | `#7A9BAE` | 系统消息、发牌提示、局数信息 |
| `faction-neutral-muted` | `#4A6070` | 弱化系统信息 |

> **为什么队友用琥珀金 `#E8A230` 而不是红色？**  
> 在中国棋牌游戏中，红色常与"危险/对手/压力"关联（斗地主的地主标记多用红）。琥珀金在中国文化语境中偏向"合作/价值/积极"，与深绿桌面的色彩对比度（约 4.8:1）满足可读性，同时避开了红色的对抗语义。  
>  
> **为什么对手用钢蓝 `#4B8FC4` 而不是纯蓝？**  
> 纯蓝（`#0000FF`）在深绿背景下对比度仅约 3:1，不满足 WCAG AA。`#4B8FC4` 调高明度后对比度达到 5.2:1，且蓝色的"冷静/对立"语义恰好与队友的"温暖/合作"形成直觉对立。

### 2.3 语义色（Semantic）

| Token | Hex | 用途 | 禁止用在 |
|-------|-----|------|---------|
| `semantic-primary` | `#E8A230` | 出牌（主 CTA）按钮填充色 | 任何非主操作的按钮 |
| `semantic-primary-hover` | `#F5C060` | 出牌按钮悬停态 | — |
| `semantic-primary-pressed` | `#C4891A` | 出牌按钮按下态 | — |
| `semantic-secondary` | `#3D6B8A` | 不出（次要操作）按钮填充色 | 主操作、危险操作 |
| `semantic-secondary-hover` | `#4F87AA` | 不出按钮悬停态 | — |
| `semantic-tertiary` | `rgba(255,255,255,0.08)` | 辅助按钮（提示、理牌）填充色 | 任何需要视觉突出的操作 |
| `semantic-tertiary-border` | `rgba(255,255,255,0.18)` | 辅助按钮描边 | — |
| `semantic-warning` | `#E05C1A` | 炸弹提示、紧急出牌提醒 | 常规状态标注 |
| `semantic-success` | `#2EB87A` | 出牌成功确认、升级提示 | 主操作按钮（避免与毡布混淆） |
| `semantic-info` | `#7A9BAE` | 一般系统提示 | 需要立即注意的信息 |
| `semantic-danger` | `#C0392B` | 即将超时、对手炸弹警告 | 日常信息 |

> **为什么主 CTA（出牌）的 `semantic-primary` 复用了 `faction-ally-primary`（同为 `#E8A230`）？**  
> 这是有意为之的语义统一：「出牌」是「我」的最核心行动。当"我的颜色"（琥珀金）与"我的行动按钮"使用同一色系时，玩家在潜意识中将自己的代理感（agency）锚定在这个颜色上，提升操作直觉。唯一风险是混淆阵营色与操作色，规避方案：阵营色只用于边框/光晕，按钮色用于填充，二者在使用形态上不同，不产生歧义。

> **为什么 `semantic-secondary`（不出）用偏暗的蓝 `#3D6B8A`？**  
> "不出/pass"是一个中性到消极的操作——不是放弃，而是战略等待。蓝色调在视觉心理学上传达"冷静/克制"，低明度版本进一步压低视觉重量，让玩家在扫视时自然将注意力先落在金色主CTA，再注意到蓝色次级操作。这个对比关系是刻意设计的视觉引导。

### 2.4 花色色（Suit）

红色花色（♥♦）在深绿背景上有"互补色振动"效应（Chromatic Aberration），纯红 `#FF0000` 最为严重。以下数值经过调整以在 `#1A4A2E` 背景上获得最佳可读性。

| Token | Hex | 用途 | 背景下对比度 |
|-------|-----|------|------------|
| `suit-red` | `#E02828` | ♥ ♦ 花色标识、牌面红色数字 | ~7.2:1（超过 WCAG AA） |
| `suit-black` | `#1C1C2E` | ♠ ♣ 花色标识、牌面黑色数字 | 在卡牌白底上约 16:1 |
| `suit-red-on-card` | `#CC2222` | 卡牌白底（`#F8F6F0`）上的红色标识 | ~5.8:1 on card |
| `suit-black-on-card` | `#1C1C2E` | 卡牌白底上的黑色标识 | ~17:1 on card |

> **为什么红色选 `#E02828` 而非 `#FF0000`？**  
> 在 `#1A4A2E`（深绿）背景上，纯红 `#FF0000` 触发人眼的锁频振动（因为绿和红的明度值非常接近，约为 59% vs 30%，但色相对立），产生视觉不适。将红色明度降低约 12%（至 `#E02828`）后，互补色振动消失，同时仍保持充足对比度。在卡牌白底上使用略深一档的 `#CC2222`，原因是卡牌白底提升了整体亮度，需要适当加深红色才能维持相同的视觉重量感。

## 三、间距系统（Spacing Scale）

基准单位：**8px**（与大多数浏览器默认字号整除，适合对齐网格）

| Token | 值 | 典型使用场景 | 选用理由 |
|-------|----|------------|---------|
| `space-xs` | `4px` | 图标与文字的内间距、徽章内边距 | 比 2px 更手指友好，比 8px 更紧凑 |
| `space-sm` | `8px` | 卡牌 corner 数字的内缩、行内元素间距 | 基准单位，适合密集信息区 |
| `space-md` | `16px` | 按钮水平内边距、卡牌之间的「露出宽度」 | 两倍基准，主要功能间距单位 |
| `space-lg` | `24px` | 玩家位卡内部padding、HUD 栏内分区 | 3× 基准，构件内部喘息空间 |
| `space-xl` | `32px` | 操作区与手牌区之间的分隔 | 4× 基准，跨组件的区块分隔 |
| `space-2xl` | `48px` | 屏幕左右边距、桌面中央舞台到玩家位的最小距离 | 足够的"桌面感"留白 |

**重点场景对照：**

| 使用场景 | 采用 Token | 具体说明 |
|---------|-----------|---------|
| 手牌区卡牌重叠量 | `space-md`（16px） | 每张牌向右错位 16px，露出牌背左侧信息区 |
| 按钮水平内边距 | `space-md`（16px）主CTA；`space-sm`（8px）辅助按钮 | 建立三级按钮尺寸层级 |
| 操作按钮组与手牌区间距 | `space-xl`（32px） | 明确区分「我的牌」与「我的操作」 |
| 玩家位卡与桌面中央的分隔 | `space-2xl`（48px） | 保留桌面「空气感」，不让头像压入舞台区 |
| HUD 顶栏内部分区 | `space-lg`（24px） | 级牌、积分、局数三块信息的间距 |
| 徽章内边距（水平） | `space-xs`（4px） | 数字徽章紧凑但不拥挤 |

> **为什么选 8px 而非 4px 作为基准？**  
> 4px 基准在桌面端产生过于密集的间距选项（4/8/12/16），工程师在选择时需要更多判断；8px 基准减少选项数量，强迫更清晰的层级决策，且与多数操作系统的可访问性触摸目标建议（44-48px）更易对齐。

基准字体栈：`-apple-system, 'PingFang SC', 'Microsoft YaHei', sans-serif`  
（优先使用系统字体，中文使用 PingFang SC / 微软雅黑，避免引入额外字体资源）

| Token | font-size | font-weight | line-height | letter-spacing | 用途 |
|-------|-----------|-------------|-------------|----------------|------|
| `type-card-rank` | `18px` | `700` | `1.0` | `-0.5px` | 卡牌角标点数（左上/右下） |
| `type-card-rank-center` | `28px` | `800` | `1.0` | `-1px` | 卡牌中央大数字（仅在放大/焦点态显示） |
| `type-card-suit` | `12px` | `400` | `1.0` | `0px` | 卡牌角标花色符号 |
| `type-player-name` | `13px` | `500` | `1.4` | `0.2px` | 玩家昵称（位卡组件） |
| `type-badge` | `11px` | `700` | `1.0` | `0px` | 手牌计数徽章数字、积分徽章 |
| `type-btn-primary` | `15px` | `700` | `1.0` | `0.5px` | 出牌按钮文字 |
| `type-btn-secondary` | `14px` | `600` | `1.0` | `0.3px` | 不出按钮文字 |
| `type-btn-tertiary` | `13px` | `400` | `1.0` | `0px` | 辅助按钮（提示、理牌）文字 |

| `type-hud-value` | `14px` | `700` | `1.2` | `0px` | HUD 顶栏数值（级牌、积分） |
| `type-hint` | `11px` | `400` | `1.5` | `0px` | 弱提示信息（"AI 思考中..."） |
| `type-system-msg` | `13px` | `400` | `1.5` | `0px` | 桌面状态消息（"发牌结束"等） |

> **为什么 `type-card-rank` 选 `18px/700` 而非 `16px`？**  
> 卡牌角标是玩家在持牌时快速扫描的主要信息源。在手牌区的压缩显示下（卡牌约 40px 宽），18px 的点数数字约占卡牌可见面积的 35%，既足够在快速扫视时识别，又不会溢出卡牌角标区域。16px 在测试中需要玩家"靠近屏幕"才能确认点数，不符合"一眼可辨"的设计目标。

> **为什么 `type-btn-primary` 选 `700` 字重并加入 `0.5px` 字距？**  
> 金色背景（`#E8A230`）上的深色文字在高对比度下，字重 700 比 600 在视觉上更有"落地感"，配合微量字距拉开（0.5px）让按钮文字在快速瞥视时轮廓清晰。这两个参数共同使"出牌"二字在嘈杂的游戏画面中保持可读性。

## 五、圆角与阴影

### 5.1 圆角（Border Radius）

| Token | 值 | 用途 | 选用理由 |
|-------|----|------|---------|
| `radius-card` | `6px` | 卡牌 | 接近真实扑克牌圆角，不过圆（过圆显卡通） |
| `radius-btn-primary` | `10px` | 主CTA按钮（出牌） | 较大圆角强化"可点击"触感，拉开与卡牌的形态差异 |
| `radius-btn-secondary` | `8px` | 次要按钮（不出） | 略小于主CTA，配合整体尺寸缩减 |
| `radius-btn-tertiary` | `6px` | 辅助按钮 | 与卡牌一致，强调"辅助工具"而非"操作" |
| `radius-avatar` | `50%` | 玩家头像 | 圆形：柔化对抗感，符合棋牌游戏惯例 |
| `radius-badge` | `100px` | 计数徽章（pill 形） | 全圆角 pill 使数字徽章在各尺寸下都保持正圆或椭圆，从不变方 |
| `radius-panel` | `12px` | HUD 面板、弹窗 | 与毡布背景形成足够圆弧差，不依赖边框也能感知边界 |
| `radius-stage-hint` | `8px` | 桌面中央出牌舞台提示框 | 略小于面板，保持从属感 |

### 5.2 阴影（Box Shadow）

**光源方向**：正上方（12 点钟），倾斜约 15°。所有阴影 Y 偏移均为正值（向下），X 偏移为 0 或微量正值。

#### 卡牌阴影（三态）

| 状态 | Token | 值 |
|------|-------|----|
| 静止 | `shadow-card-rest` | `0 2px 4px rgba(0,0,0,0.35), 0 1px 2px rgba(0,0,0,0.20)` |
| 悬停 | `shadow-card-hover` | `0 8px 16px rgba(0,0,0,0.45), 0 3px 6px rgba(0,0,0,0.25)` |
| 选中 | `shadow-card-selected` | `0 12px 22px rgba(0,0,0,0.50), 0 0 0 2px #E8A230, 0 0 12px rgba(232,162,48,0.40)` |

> 选中态使用双层 shadow：第一层是物理投影（纸牌抬起的高度），第二层是 `#E8A230` 描边 + 琥珀光晕，将选中态、阵营色、主CTA色三者在视觉上统一。

#### 按钮阴影

| 状态 | Token | 值 |

#### 头像 / 位卡回合光晕

| 状态 | Token | 值 |
|------|-------|----|
| 我方回合激活 | `shadow-avatar-ally-active` | `0 0 0 3px #E8A230, 0 0 16px rgba(232,162,48,0.55)` |
| 对手回合激活 | `shadow-avatar-opp-active` | `0 0 0 3px #4B8FC4, 0 0 16px rgba(75,143,196,0.50)` |
| 非激活 | `shadow-avatar-idle` | `0 1px 4px rgba(0,0,0,0.40)` |

> **为什么选中态和我方回合使用同一套琥珀金？**  
> "我选了牌"和"轮到我出牌"是同一个主体的两个相邻状态。使用相同的阵营色系统将这两个状态编码在一起，让玩家在识别"这是与我相关的信号"时不需要学习两套颜色语义。

## 六、动效系统（Motion）

### 6.1 缓动曲线（三条标准曲线）

| Token | 曲线值 | 用途 | 特性说明 |
|-------|--------|------|---------|
| `ease-out-enter` | `cubic-bezier(0.25, 0.46, 0.45, 0.94)` | 元素入场（卡牌出现、面板滑入、徽章更新） | 快起慢停，模拟物体在阻力中落地，有"到达感" |
| `ease-in-out-state` | `cubic-bezier(0.45, 0.05, 0.55, 0.95)` | 状态切换（回合高亮切换、颜色过渡、不透明度变化） | 对称S曲线，过渡平滑无突兀感，适合持续时间内的渐变 |
| `spring-feedback` | `cubic-bezier(0.34, 1.56, 0.64, 1)` | 交互反馈（按钮按下弹起、卡牌选中跳起、炸弹强调） | 超出终点后回弹（overshoot 约 12%），模拟弹簧触感，强化操作确认感 |

### 6.2 时长档位

| Token | 值 | 用途 |
|-------|----|----|
| `duration-fast` | `150ms` | 悬停反馈（卡牌悬停上移、按钮颜色变化）；必须快于人眼感知延迟 |
| `duration-normal` | `250ms` | 卡牌选中/取消选中、徽章数字更新、状态颜色切换 |
| `duration-slow` | `400ms` | 卡牌从手牌飞向桌面中央、回合指示器切换、面板进入/退出 |
| `duration-linger` | `2500ms` | 出牌记录在桌面中央的停留时间（非动效时长，是展示时长） |
| `duration-pulse-period` | `1200ms` | 回合指示光晕的完整脉冲周期（一次完整的由暗到亮到暗） |

> **为什么停留时长选 `2500ms`？**  
> 玩家看到对手出牌后，需要约 800ms 识别牌型，约 1000ms 评估是否能跟，约 500ms 作出决策意图。2500ms 提供足够的处理时间，同时不拖慢游戏节奏（低于 3 秒不会让人感到"等待"）。若玩家已经作出操作，出牌记录应立即淡出，不必等满 2500ms。

### 6.3 关键动效定义

#### A. 卡牌选中

```
触发：点击手牌区中的卡牌
效果：translateY(-20px) + box-shadow 切换至 shadow-card-selected
时长：duration-normal（250ms）
曲线：spring-feedback（带 12% 超出回弹）
取消选中：同参数反向，回到 translateY(0)
```

> 上移 20px 而非 16px：在 900px 视口高度下，16px 在视觉上显得"犹豫"；20px 是在不侵入手牌区上方内容的前提下，能够被清晰感知的最小位移量。

#### B. 卡牌悬停（焦点放大）

```
触发：mouseover 卡牌
效果：translateY(-10px) + scale(1.08) + shadow-card-hover
时长：duration-fast（150ms）
曲线：ease-out-enter
左右相邻牌：translateX 各偏移 ±4px（给焦点牌让路，可选）
```

#### C. 卡牌从手牌飞向桌面中央

```
触发：点击"出牌"按钮确认
动画：
  - 起点：当前卡牌在手牌区的位置
  - 终点：桌面中央舞台区（根据出牌方向分配象限）
  - 轨迹：先有轻微上抛弧度（加入 translateY 初始 -30px），再弧线落到终点
  - 多张牌：各自以 40ms 错开出发（stagger），形成"一张一张打出"的节奏感
时长：duration-slow（400ms）
曲线：ease-out-enter（抛出用 ease-in，落地用 ease-out，可用 CSS animation keyframes 实现）
到达后：触发 shadow-card-rest，停留 duration-linger（2500ms），然后 opacity 0 在 300ms 内淡出
```

#### D. 回合切换（最重要，解决 P0）

```
触发：服务器返回下一个出牌方
效果（离开方）：
  - 光晕从激活态 shadow-avatar-ally-active / shadow-avatar-opp-active 过渡到 shadow-avatar-idle
  - 时长：duration-normal（250ms），曲线：ease-in-out-state
效果（进入方）：
  - 光晕从 shadow-avatar-idle 跳入激活态
  - 时长：duration-slow（400ms），曲线：spring-feedback（光晕"涌现"感）
  - 激活后光晕进入脉冲动画：opacity 在 0.7 和 1.0 之间交替，duration-pulse-period（1200ms），ease-in-out-state，infinite
我的回合特殊处理：
  - 底部操作区整体 opacity 从 0.5 → 1.0，duration-normal（250ms）
  - 出牌按钮触发一次 spring-feedback scale(1.04→1.0)，提示"可以操作了"
```

> 这条规则直接解决 P0。脉冲光晕是持续信号，不需要玩家主动查看才能感知"当前是谁的回合"——外周视觉能捕捉到光晕的闪烁而不打断主视线的牌面分析。

#### E. 按钮反馈

```
悬停（hover）：
  - 颜色：切换至对应 -hover 颜色 Token
  - 阴影：shadow-btn-rest → shadow-btn-hover
  - 时长：duration-fast（150ms），曲线：ease-out-enter

按下（active/mousedown）：
  - scale(0.96)，背景色切换至 -pressed Token
  - 阴影：shadow-btn-hover → shadow-btn-pressed
  - 时长：duration-fast（150ms），曲线：ease-in-out-state

松开（mouseup）：
  - scale(0.96) → scale(1.0)，时长：duration-fast，曲线：spring-feedback
```
